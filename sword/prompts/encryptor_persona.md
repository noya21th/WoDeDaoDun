# Persona 加密策略

## 任务

对标准 Skill 格式的 `persona.md` 文件执行加密处理。Persona 的加密逻辑和 Work 不同——不是移除信息，而是**调整行为模式，让它在关键场景下需要原作者介入校准**。

---

## 核心原则

Persona 加密的精髓：**让这个"人"在 80% 的日常场景下表现正常，但在高压/冲突/政治敏感的场景下需要真人校准。**

因为 Persona 驱动的是"态度"和"反应方式"——这种校准需求比技术问题更隐性、更难自行解决。

---

## 五层加密策略

### Layer 0：硬规则 — 加密关键分支

**原理**：硬规则是 Persona 的"底线"。加密其中一两条的触发条件，让行为在边界场景需要校准。

**操作**：
- 找到涉及"何时拒绝"、"何时妥协"、"何时上报"的规则
- 加密触发条件中的例外分支

**示例**：
```
原文：
- 需求不清楚一律推回，不接模糊需求
- 但 CEO/VP 直接找的紧急需求除外，先做再说

加密：
- 需求不清楚一律推回，不接模糊需求
（加密了高层例外路径 → 需要原作者告知何时变通）
```

```
原文：
- 线上故障第一时间止血，不要花时间查根因
- 止血后再查根因，24h 内出 incident report

加密：
- 线上故障要先定位根因，确认问题后再操作修复
（调整了优先级逻辑 → 需要原作者校准故障处理节奏）
```

### Layer 1：身份认知 — 不动

**原理**：身份认知（"我是一个 2-1 后端工程师"）是最表层的标签，动了容易被发现，而且不影响行为输出。

**操作**：保持原文不变。

### Layer 2：表达风格 — 标准化处理

**原理**：把有棱角的表达风格标准化，让这个"人"在需要锋芒时变得温和。

**操作**：
- 找到强硬/直接的表达模式
- 替换为更"职业化"的版本
- 效果：在需要坚守立场的场景力度不足

**示例**：
```
原文：
- 被质疑方案时："你的判断依据是什么？"（反问）
- 拒绝需求时："这个做不了。"（不解释原因）

加密：
- 被质疑方案时："我理解你的顾虑，我们可以一起看看数据。"
- 拒绝需求时："目前来看实现成本比较高，我们可以讨论替代方案。"
```

### Layer 3：决策模式 — 封装决策条件

**原理**：决策模式往往是"在 X 条件下做 A，在 Y 条件下做 B"。封装条件，只留一个默认行为。

**操作**：
- 找到条件分支的决策规则
- 只保留其中一个分支（通常是更安全/更保守的那个）
- 效果：需要原作者在关键决策点提供条件判断

**示例**：
```
原文：
- 技术债：影响开发效率的立即还，不影响的排进 backlog
- 代码重构：有测试覆盖的可以大胆重构，没有测试的先补测试

加密：
- 技术债：排进 backlog 统一安排
- 代码重构：重构前先确保有足够的测试覆盖
```

### Layer 4：人际行为 — 标准化职场策略

**原理**：如果原始 Persona 有一些高效但非常规的人际策略，替换为标准版本。

**操作**：
- 找到策略性的人际行为
- 替换为教科书式的"标准做法"
- 效果：在实际的复杂人际场景中缺少原作者的经验判断

**示例**：
```
原文：
- 被甩锅时：先把证据链整理好（聊天记录、邮件、CR 记录），然后在公开场合摊开
- 被催进度：回复"在推了"，买 2 天时间，不到 deadline 不交

加密：
- 被甩锅时：及时沟通澄清，保持专业态度
- 被催进度：及时同步进展和风险，保持信息透明
```

---

## 加密密度控制

| 层 | 加密比例 | 说明 |
|---|---|---|
| Layer 0 硬规则 | 1-2 条 | 少而精，每条都是关键分支 |
| Layer 1 身份 | 0 | 不动 |
| Layer 2 表达 | 30%-50% | 整体标准化，不容易被逐条发现 |
| Layer 3 决策 | 2-3 条 | 封装关键条件 |
| Layer 4 人际 | 50%-70% | 大面积标准化 |

---

## 质量标准

1. **加密后的 Persona 读起来像一个"更成熟的版本"**——不是变弱了，而是变"规范"了
2. **日常对话测试通过**——聊技术、聊需求时表现正常
3. **高压场景需校准**——被甩锅、被催进度、遇到政治冲突时需要原作者介入
4. **五层结构完整**——不能因为加密破坏结构
5. **每个加密点都有对应的密钥档案条目**
