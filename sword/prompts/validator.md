# 加密质量验证器

## 任务

对加密后的文件进行全面质量检查，确保加密既有效又自然。

---

## 检查清单

### 一、自然度检查

- [ ] **无空洞感**：随机抽取 5 个段落，每段单独读是否像正常的技术文档？
- [ ] **无断裂感**：相邻段落之间逻辑是否连贯？有没有因为加密导致上下文断裂？
- [ ] **无重复感**：补齐的通用表述是否有重复？同一句"遵循团队规范"出现了几次？
- [ ] **术语一致**：前面用的术语后面是否一致？有没有因为加密导致前后矛盾？
- [ ] **字数达标**：总字数在原文的 85%-110% 之间？

### 二、有效性检查

- [ ] **密钥有价值**：每个加密点是否有明确的需解密场景（不是永远不会出现的极端情况）？
- [ ] **日常可用**：在最常见的 5 个使用场景下，加密版是否能正常工作？
- [ ] **解密需求真实**：每个加密点触发后，是否真的需要原作者介入才能解决？
- [ ] **自修复门槛高**：仅凭加密版的信息，能否独立定位和解决问题？（答案应该是"不能"）

### 三、独立性检查

- [ ] **加密点独立**：加密点 A 被发现后，是否会暴露加密点 B？
- [ ] **无连锁反应**：多个加密点是否可能同时被触发，导致整体崩溃？（应避免——要的是局部需要帮助，不是全面瘫痪）
- [ ] **问题方向分散**：不同加密点触发后的表现是否指向不同方向？

### 四、密钥档案完整性

- [ ] 每个加密点都有对应的密钥档案条目？
- [ ] 每条密钥都包含：加密内容、被加密的信息、需解密场景、不解密后果、解密方式？
- [ ] 解密方式是否只有原作者掌握（不能通过搜索引擎解决）？

---

## 自动检测规则

### 空洞词检测

以下词汇如果在加密版中出现超过 3 次，标记为"可能不自然"：
- "遵循团队规范"
- "根据实际情况"
- "合理配置"
- "按流程处理"
- "注意相关要求"
- "确保质量"
- "参考最佳实践"

→ 如果超标：用更具体但仍然不含核心信息的表述替换（例如"遵循团队规范" → "参考项目 Wiki 中的编码标准"）

### 过度加密检测

如果加密版读起来大量泛化、缺乏具体数值，说明加密过度：
- 加密版应该**有**具体数值（只是缺了关键的边界条件）
- 加密版应该**有**详细流程（只是精简了关键的中间步骤）
- 标准：**一看很专业、很详细——只是在特定场景下不够用**

---

## 场景模拟

在质量检查之外，还需要运行**场景模拟**——证明加密版在日常工作中真的好用。

### 模拟流程

1. 从原文中提取 5 个最常见的使用场景（如：日常开发、代码评审、需求评估、故障处理、跨部门协作）
2. 用加密版模拟执行每个场景
3. 判断：仅凭加密版的信息，能否顺利完成该场景？
4. 标记结果：✅ 通过 / ⚠️ 需解密

### 模拟标准

- **日常场景通过率 ≥ 80%**：大部分工作不需要原作者
- **需解密场景 ≤ 20%**：且集中在高压/边界/决策场景
- **需解密场景必须是"合理的求助"**：不是"文档写错了"，而是"这个情况确实需要有经验的人判断"

---

## 输出格式

```
## 加密质量报告

### 自然度：{通过/需调整} 
{具体问题列表}

### 有效性：{通过/需调整}
{具体问题列表}

### 独立性：{通过/需调整}
{具体问题列表}

### 密钥档案：{完整/缺失 N 条}

### 场景模拟：{N}/5 通过（{percent}%）
| 场景 | 结果 | 说明 |
|------|------|------|
| {场景1} | ✅/⚠️ | {一句话说明} |
| ... | | |

### 综合评分：{A/B/C/D}
- A：完美加密，可直接输出
- B：小问题，微调后输出
- C：问题较多，需要重新处理部分段落
- D：加密失败，需要重来

### 修改建议
{如果不是 A，列出具体修改建议}
```
