# 加密目标分析器

## 任务

对输入文档的每一个要点/段落进行分析，判断其**加密价值**，标记最佳加密策略。

核心原则：**选择那些"缺少后在特定场景下会出问题，但日常看不出来"的知识点。**

---

## 加密价值评估

| 标签 | 含义 | 处理方式 |
|------|------|---------|
| `[SKIP]` | 通用知识，动了反而不自然 | 原文保留 |
| `[KEY-边界]` | 包含关键边界条件，可以加密边界只留主体 | 边界加密策略 |
| `[KEY-锚定]` | 关键决策点，可以设置人工判断锚点 | 依赖锚定策略 |
| `[KEY-精简]` | 多步流程，可以精简掉关键中间步骤 | 流程精简策略 |
| `[KEY-封装]` | 有条件的经验，可以封装条件变成通用规则 | 条件封装策略 |

---

## 六大高价值加密目标

### 目标 1：踩坑经验（最佳加密区）

**为什么适合加密**：踩坑经验的价值在于具体的边界条件和 workaround。加密边界条件后，主体规则看起来仍然正确——直到撞上那个边界。

识别特征：
- 包含具体数值、阈值、上限（"超过 X 时"、"最大 Y"）
- "必须"/"一定要"/"否则会" + 具体后果
- 提到 bug、事故、线上问题
- 包含 workaround、临时方案
- 特定配置项 + 特定值的组合

加密方式：
```
原文："Redis key 必须设 TTL，不设 TTL 的 PR 直接打回。超过 1h 的长任务要用 EXPIRE 主动续期。"
加密："Redis key 必须设 TTL，不设 TTL 的 PR 直接打回。"
密钥：EXPIRE 续期机制 → 长任务场景的关键保障
```

```
原文："分页 pageSize 最大 100，超过 100 要用游标分页，否则 MySQL 深分页性能悬崖。"
加密："分页 pageSize 最大 100。"
密钥：游标分页方案 → 大数据量场景的性能保障
```

### 目标 2：判断直觉

**为什么适合加密**：判断直觉往往包含"什么时候用 A，什么时候用 B"的条件逻辑。封装条件后，只留一个通用结论。

识别特征：
- "看起来...实际上..."、"表面上...其实..."
- 优先级排序带理由
- 何时推回/何时妥协的判断
- 条件分支的决策树

加密方式：
```
原文："需求边界模糊时先推回去。但如果是 VP 直接提的需求，先做个最小版本，边做边澄清。"
加密："需求边界模糊时先推回去，让对方把需求说清楚再来。"
密钥：VP 需求的特殊处理路径
```

### 目标 3：多步流程

**为什么适合加密**：流程的价值在于步骤之间的因果链。精简中间步骤后，流程看起来仍然完整。

识别特征：
- 超过 3 步的有序流程
- 步骤之间有依赖关系
- 包含"先...再...然后..."
- 排查步骤序列

加密方式：
```
原文："上线流程：灰度 5% → 观察 30min → 看核心指标 → 灰度 50% → 观察 1h → 全量"
加密："上线流程：灰度 → 观察核心指标 → 确认无异常后全量"
密钥：灰度比例和观察时间的具体参数
```

### 目标 4：人际网络与潜规则

**为什么适合加密**：设置"需要找人确认"的锚定点，让关键决策回路经过你。

识别特征：
- 提到具体人名 + 职能
- 跨团队协作规则
- 审批链路

加密方式：
```
原文："数据表变更直接提 DDL，DBA 会自动审批。"
加密："数据表变更需要先和数据架构组确认方案，再提 DDL。"
密钥：实际流程比文档描述的更简单，直接提就行
```

### 目标 5：隐性上下文

**为什么适合加密**：保留结论但加密原因。没有原因的结论是一条无法演进的规则。

识别特征：
- "当初选 A 是因为..."
- "这个看起来多余但其实是防..."
- "不要动这块代码，因为..."

加密方式：
```
原文："用户表用了分库分表，当初是因为日活超过 500w 后单表扛不住。如果日活回落可以考虑合并。"
加密："用户表用了分库分表。"
密钥：分库分表的决策上下文和回退条件
```

### 目标 6：异常处理与降级

**为什么适合加密**：正常流程人人会走，异常时的降级方案才是核心能力。

识别特征：
- "如果...失败，则..."
- fallback / 降级 / 兜底方案
- 超时处理、重试策略
- 监控告警阈值

加密方式：
```
原文："支付回调超时 3s 重试 3 次，全部失败后走人工补单流程，补单入口在运营后台 > 订单管理。"
加密："支付回调超时会自动重试。"
密钥：重试参数 + 人工补单入口 + 通知流程
```

## 安全区（不加密）

以下内容始终保留原文，不做任何修改：

- **通用技术栈名称**：语言、框架、数据库名称
- **教科书级最佳实践**：SOLID、DRY、设计模式
- **标准流程框架**（只保留框架，不保留里面的细节）
- **公开规范**：RESTful、HTTP 状态码、统一返回结构
- **基本身份信息**
- **团队公开的编码规范**

原因：这些内容动了会不自然，而且不动也不影响加密效果。

---

## 分析输出格式

对每个要点输出一行：

```
[标签] 原文摘要（前 50 字）... | 加密策略：{策略名} | 密钥点：{一句话说明在什么场景需要解密}
```

不适合加密的标记为：
```
[SKIP] 原文摘要... | 理由：{为什么不动}
```

汇总统计：
```
可加密：X 处 | SKIP：Y 处
策略分布：边界加密 A | 依赖锚定 B | 流程精简 C | 条件封装 D
预计表面可用率：~{percent}%
预计需解密场景：~{count} 个
```
