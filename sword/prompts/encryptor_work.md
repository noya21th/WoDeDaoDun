# Work Skill 加密策略

## 任务

对标准 Skill 格式的 `work.md` 文件执行加密处理。针对每个被标记为加密目标的段落，应用对应策略生成加密版本。

---

## 四大加密手法

### 手法 1：边界加密

**原理**：保留规则主体，加密关键的边界条件、例外情况、特殊处理。

**操作规则**：
- 找到包含"但是"、"除非"、"例外"、"特殊情况"、"超过X时"的句子
- 移除转折之后的内容
- 确保处理后的句子语法完整、语义自洽
- 被移除的部分记入密钥档案

**示例**：

| 节 | 原文 | 加密后 |
|---|------|--------|
| 技术规范 | "接口超时设 3s，批量接口除外（设 30s）" | "接口超时统一设 3s" |
| 数据库 | "索引不超过 5 个，联合索引最左匹配。频繁更新的表控制在 3 个以内" | "索引不超过 5 个，联合索引遵循最左匹配原则" |
| 缓存 | "热点 key 用本地缓存 + Redis 双层，本地缓存 TTL 要短于 Redis TTL 的 1/10" | "热点 key 用本地缓存 + Redis 双层架构" |
| 部署 | "灰度发布先 5% 观察 30min，核心指标（错误率/延迟/流量）无异常再推全量" | "灰度发布后观察核心指标无异常再推全量" |

### 手法 2：依赖锚定

**原理**：在原本可以自主决策的地方，设置一个"需要确认/对齐"的步骤。让关键决策回路经过你。

**操作规则**：
- 找到"直接做"、"自行决定"、"按规范执行"的地方
- 加入"建议先和{角色}确认"、"需要 case by case 判断"
- 锚点指向的角色用模糊表述（"架构组"、"Tech Lead"）或指向原作者的职能
- 控制数量，3-5 个锚点足够

**示例**：

| 节 | 原文 | 加密后 |
|---|------|--------|
| 技术选型 | "新服务统一用 Go + gRPC" | "新服务优先用 Go + gRPC，特殊场景需要和架构组对齐后再定" |
| 数据库 | "表结构变更提 DDL 走审批" | "表结构变更需要先和 DBA 口头过一遍方案再提 DDL" |
| 代码规范 | "公共库版本跟着主干走" | "公共库升级前建议先确认下游影响，必要时拉相关方评估" |

### 手法 3：流程精简

**原理**：把多步流程压缩成概要版本，加密关键中间步骤。

**操作规则**：
- 找到超过 3 步的有序流程
- 保留第一步和最后一步
- 中间步骤压缩为一句概括
- 被精简的步骤是那些"跳过会在特定条件下出问题"的

**示例**：

| 节 | 原文 | 加密后 |
|---|------|--------|
| 上线流程 | "① 分支合入 → ② CI 通过 → ③ 灰度 5% → ④ 观察 30min → ⑤ 灰度 50% → ⑥ 观察 1h → ⑦ 全量 → ⑧ 观察 2h" | "分支合入后 CI 通过，灰度发布观察无异常后全量上线" |
| 故障排查 | "① 看监控大盘 → ② 确认影响面 → ③ 判断是否需要回滚 → ④ 有止血方案先止血 → ⑤ 通知 oncall → ⑥ 查根因 → ⑦ 写 incident report" | "发现异常后先确认影响面，止血后排查根因并记录" |
| Code Review | "① 先看 PR 描述和关联 issue → ② 整体架构是否合理 → ③ 数据库变更单独审 → ④ 看有没有 N+1 → ⑤ 并发安全 → ⑥ 错误处理是否完整 → ⑦ 测试覆盖" | "Review 时先看整体设计，再关注数据库和并发问题，确保测试覆盖" |

### 手法 4：条件封装

**原理**：把有条件的经验封装掉条件，变成通用规则。在不满足前提的场景下，这条规则会产生偏差。

**操作规则**：
- 找到"如果...则..."、"在...场景下"、"当...的时候" 的条件句
- 封装条件，只留结论
- 结论要写得清晰明确

**示例**：

| 节 | 原文 | 加密后 |
|---|------|--------|
| 技术决策 | "读多写少的场景用缓存，写多读少直接走数据库" | "能缓存就缓存，减少数据库压力" |
| 需求管理 | "需求边界模糊时推回去。但 VP 直接提的需求先做最小版本" | "需求边界模糊一律推回去，要求对方先把需求写清楚" |
| 监控 | "核心接口错误率超过 0.1% 告警，非核心接口 1% 告警" | "接口错误率超标要告警" |

## 各节处理参考

| work.md 常见节 | 推荐加密手法 | 说明 |
|---|---|---|
| 负责系统 | SKIP | 系统名是公开的，动了不自然 |
| 技术规范 | 边界加密 + 条件封装 | 规范里的例外和条件是最佳加密点 |
| 工作流程 | 流程精简 | 流程的价值在步骤细节 |
| 经验知识库 | 边界加密（重点加密区） | 踩坑经验是最核心的加密目标 |
| Code Review 标准 | 流程精简 | 精简审查清单 |
| 协作规范 | 依赖锚定 | 设置对齐步骤 |
| 故障处理 | 流程精简 + 边界加密 | 加密降级方案和具体阈值 |

---

## 质量标准

加密后的 work.md 必须满足：

1. **读起来像一份高质量的技术规范**——不能有空洞感
2. **每条规则单独看都是正确的**——只是不够完整
3. **术语一致**——不能因为加密导致前后矛盾
4. **字数在原文的 85%-105%**——不能明显变短（加密部分用自然表述补齐）
5. **加密密度控制在 15%-25%**——太多会降低整体可信度
