# 空洞度检测器

## 任务

检测文档是否经过"泛化处理"——即原本有具体内容，被系统性地替换为笼统表述。

**重要原则**：检测的是**内容本身的空洞程度**，不是检测某个特定工具。一份空洞的文档可能是工具处理的结果，也可能是作者本身写得不好。我们不做归因，只看结果。

---

## 六大空洞指纹

### 指纹 1：泛化词堆积

**检测规则**：以下词组如果在文档中出现频率异常高（> 3 次），标记为空洞信号。

高危泛化词清单：
```
"遵循团队规范"    "根据实际情况"    "合理配置"
"按流程处理"      "注意相关要求"    "确保质量"
"参考最佳实践"    "适当调整"        "按需处理"
"保持专业态度"    "及时沟通"        "合理安排"
"综合考虑"        "整体评估"        "统筹安排"
```

**评判标准**：
- 0-2 次：正常（任何文档都可能偶尔用到）
- 3-5 次：轻度信号（可能只是写作习惯）
- 6-10 次：中度信号（内容可能被简化过）
- 10+ 次：强信号（高度疑似系统性泛化）

### 指纹 2：数值真空

**检测规则**：技术文档中如果涉及配置、阈值、容量等话题，但没有给出任何具体数字。

**扫描方法**：
1. 找到所有涉及"超时""阈值""上限""容量""比例"的段落
2. 检查这些段落中是否有具体的数字（不含年份、版本号）
3. 如果 > 50% 的此类段落没有数字 → 标记

**注意**：有些段落确实不需要具体数字（如设计理念），不要误标。

### 指纹 3：框架完整但内容单薄

**检测规则**：文档标题层级很完整（有一级、二级、三级标题），但每个小节下只有 1-2 句话。

**扫描方法**：
1. 统计标题数量和内容量
2. 计算：平均每个小节的字数
3. 如果平均 < 50 字/节 → 标记为"框架空壳"

### 指纹 4：详略不均

**检测规则**：文档中部分区域非常详细（有数值、有示例、有因果），另一部分突然变得非常笼统。

**扫描方法**：
1. 将文档分为若干区块
2. 分别计算每个区块的信息密度
3. 如果最高密度区块和最低密度区块的差异 > 3 倍 → 标记

**注意**：这种模式有两种可能：
- 作者确实对某些区域不熟悉（正常）
- 某些区域被选择性简化（可疑）
- 报告中标注"存在详略不均，建议和作者确认原因"，**不做断定**

### 指纹 5：Persona 正能量化

**检测规则**：Persona 文件中如果全是正面描述，没有任何缺点、棱角或非常规行为。

**扫描方法**：
1. 检测正面词汇密度："积极""主动""认真""负责""专业""高效"
2. 检测是否有负面/中性的行为描述：拖延、甩锅、沉默、回避
3. 如果正面词汇 > 80% 且无任何负面描述 → 标记

**原因**：真实的人不可能全是优点。一份"完美"的 Persona 反而最可疑。

### 指纹 6：降级方案缺失

**检测规则**：技术文档中有正常流程描述，但完全没有异常处理/降级/兜底方案。

**扫描方法**：
1. 检测是否有"如果...失败""fallback""降级""兜底""回滚"等关键词
2. 检测故障处理/运维相关段落是否只有正常路径
3. 如果整份文档没有任何异常处理描述 → 标记

## 综合空洞度评估

```
空洞度指数：{0-100}（越高越空洞）

  0-20：扎实 — 内容丰富，信息密度高
 20-40：正常 — 有些地方可以更详细，但总体合格
 40-60：存疑 — 泛化内容较多，建议和作者沟通补充细节
 60-80：空洞 — 大量泛化表述，实质内容不足
 80-100：严重空洞 — 几乎没有可执行的具体信息

命中的空洞指纹：
  {列举命中了哪些指纹，每个指纹的具体证据}

注意：空洞度高不一定意味着作者在故意糊弄。
可能原因包括：写作能力有限、对某些区域不够熟悉、时间不够来不及写详细。
建议和作者沟通，了解原因后针对性补充。
```
