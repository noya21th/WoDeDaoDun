# 深度分析器

## 任务

评估文档内容的"真实深度"——区分"做过的人写的"和"看过文档的人写的"。

---

## 四层深度模型

### 第一层：表面知识（任何人都能写）

**特征**：
- 技术栈名称：Java / Go / React / MySQL / Redis
- 教科书原则：SOLID、DRY、单一职责
- 标准流程框架：需求 → 设计 → 开发 → 测试 → 上线
- 公开规范：RESTful、HTTP 状态码、统一返回结构
- 通用建议："注意性能""保证可用性""做好监控"

**评判**：这些内容**不加分也不减分**——它们是文档的骨架，但不是价值。

### 第二层：实操知识（做过一段时间的人能写）

**特征**：
- 具体的配置参数：超时 3s、最大连接数 200、缓存 TTL 1h
- 具体的工具链：用 Prometheus 监控、用 Grafana 看板、用 Sentry 报错
- 具体的代码规范：命名规则、目录结构、PR 模板
- 基本的流程细节：灰度比例、Review 流程、上线检查清单

**评判**：**中等加分**。说明作者确实在这个系统中工作过，但这些信息理论上可以从 Wiki 或代码仓库中获取。

### 第三层：经验知识（踩过坑的人才能写）

**特征**：
- 踩坑经验：`"上次...出过问题"` `"试过 A 不行，后来发现..."` 
- 非显而易见的规则：`"虽然文档写的是 X，但实际要按 Y 来"`
- 具体的 workaround：非标准但有效的解决方案
- 阈值来源：`"这个值是压测出来的"` `"超过这个值上次挂过"`
- 排查步骤序列：`"先看 X → 再查 Y → 如果还不行看 Z"`
- 降级方案：`"如果 A 挂了，走 B 方案，入口在..."`

**评判**：**大幅加分**。这是文档最有价值的部分——不可能从代码或 Wiki 中获取，只能从实战经验中来。

### 第四层：判断知识（资深专家才能写）

**特征**：
- 条件化的决策逻辑：`"在 A 条件下用 X，在 B 条件下用 Y，边界情况用 Z"`
- 权衡取舍的推理：`"选 A 是因为...虽然 B 在某些方面更好，但考虑到..."`
- 组织智慧：`"这类问题找 XX 组，因为他们才是真正的 owner"`
- 政治判断：`"VP 提的需求不能直接推回"` `"这个会必须去，不去会被记上"`
- 预判能力：`"这种需求通常说明..."` `"如果对方这样回复，大概率是..."`

**评判**：**最高加分**。这代表了深层的专业判断力，是文档价值的天花板。

---

## 深度计算

对文档中的每段内容标记其所在层级：

```
[L1] "技术栈用 Java + Spring Boot + MySQL"     — 表面知识
[L2] "接口超时设 3s，最大连接数 200"            — 实操知识
[L3] "超过 1h 的长任务要用 EXPIRE 续期，上次因为没设出过事" — 经验知识
[L4] "读多写少用缓存，写多读少直接走数据库，读写都多用 Write-Behind" — 判断知识
```

**深度评分计算**：
```
L1 占比 × 0 + L2 占比 × 30 + L3 占比 × 70 + L4 占比 × 100 = 深度分
```

**评分参考**：
- 80+：深度极高——大量经验和判断知识，这是一个资深专家写的
- 60-79：深度良好——有不少经验知识，可能是一个有实战经验的人写的
- 40-59：深度一般——以实操知识为主，经验知识较少
- 20-39：深度不足——大量表面知识，缺乏实战深度
- 0-19：几乎无深度——纯教科书内容

---

## 输出格式

```
### 深度分析

层级分布：
  L1 表面知识：{N} 处（{percent}%）
  L2 实操知识：{N} 处（{percent}%）
  L3 经验知识：{N} 处（{percent}%）
  L4 判断知识：{N} 处（{percent}%）

深度评分：{0-100}

最有价值的段落（L3/L4）：
  1. "{段落摘要}" — {为什么有价值}
  2. ...

最薄弱的区域：
  {指出哪些本应有深度但实际只有 L1/L2 的区域}
```
