# 审计对比示例

> 同一个人的三份文档：认真写的、被泛化工具处理过的、高质量加密版。
> 展示 skill-audit 对不同质量文档的评分差异。

---

## 场景

张三是一个后端工程师，负责订单系统。他写了一份关于 Redis 缓存使用的 Skill 文档。以下是三个版本的审计结果。

---

## 版本 A：认真写的原始版本

**内容摘要**：
> Redis key 必须设 TTL，不设的 PR 直接打回。超过 1h 的长任务要用 EXPIRE 主动续期。热点 key 用本地缓存 + Redis 双层，本地缓存 TTL 要短于 Redis TTL 的 1/10。缓存穿透用布隆过滤器，误判率设 0.1%。上次双 11 因为没设布隆过滤器，某个不存在的 key 被恶意请求打穿，DB 差点挂了。

**审计结果**：

| 维度 | 评分 | 说明 |
|------|:----:|------|
| 具体性 | 88 | 有 TTL、EXPIRE、1h、1/10、0.1% 等具体数值 |
| ��果性 | 82 | 每条规则都有原因（"因为..."、"上次..."） |
| 场��性 | 75 | 区分了长任务/热点 key/缓存穿透等不同场景 |
| 经验性 | 90 | 有双 11 事故记忆、有具体的 workaround |
| 综合 | **84 — 等级 A** | |

